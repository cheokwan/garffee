1. Feature Overview and Description

Garffee iOS is the mobile client complementing the whole “Smart Cafe” project which involves a physical tech-integrated coffee shop, a backend order scheduling system and API connecting the store and the app, and the app itself which allows user to pre-order on the go.

3. Design Goals and Non-Goals

The original focus of the app is to provide an intuitive user interface and location tracking ability to ensure a high quality user experience and a just-in-time delivery of the order. The design principle is similar to other typical business-oriented mobile clients which is to keep a fast iteration and responsive to changes of product decisions. Hence the philosophy is to keep the design simple and prefer convention of programming patterns over configurations.

        The front end UI design is driven by a wireframe separately documented[1]. The interface design is targeted to usability and a smooth ordering process. On the other hand, the backend design of the app consists of three main parts: a clearly-defined model, controller, and view layers; a reliable component for communicating with a RESTful backend web service API[2] and model[3]; and pushing the limits of location tracking on iOS to provide a better arrival time estimation for the user in order to drive the fresh delivery of the order.

4. Dependencies

The native iOS and Objective-C patterns, conventions, and guidelines are followed as much as possible while building the app for maximum support and maintainability. Native Apple provided frameworks are used throughout the app, such as:

  • CoreData - the versatile object graph and relational data persistence framework.
  • CoreLocation - the location tracking framework combining GPS, Wifi-fingerprinting, and Cellular triangulation technology.
  • AudioToolbox and Accelerate - for recording audio data, processing and analyzing the sound wave data with hardware accelerated library. Used for high frequency sound wave detection for a novel take on the location tracking problem.
  • CoreBluetooth - for incorporating the new Bluetooth low-energy iBeacon technology used for short distance location tracking.
  • Security - for securely storing/retrieving sensitive user data with Apple’s Keychain service.
  • MapKit - for displaying maps showing store location with Apple’s map API and data.
  • AddressBook - for retrieving user’s local phone contacts information.

A number of third-party libraries are included to facilitate different functionalities of the product. Below is a list of all the libraries used, their licenses, descriptions, and justifications:

  • ECSlidingViewController[MIT][7] - view controller container that provides the slide menu effect.

5. Requirements, Assumptions and Constraints

Most features of the mobile client are driven by the UI/UX design in the wireframe in footnote #1 and should be quite self explanatory. The detailed product decisions and requirements are done mostly through verbal communications and not documented. The design is focused on iOS7 with support of both 3.5” and 4” screens in portrait orientation. Below is an overview of the main product features in the app:

  • Facebook login integration and bootstrap app user management.
  • Promotional daily award game and sponsor ad displaying.
  • Fluid ordering page based on dynamic products and options menu fetched from backend.
  • Purchasing gift orders for friends discovered from Facebook or local phone contacts.
  • Shopping cart with recipient selection and editing ability, store selection and checkout.
  • User location data reporting to backend based on the array of location tracking services available.
  • Location tracking layer combining CoreLocation, high frequency sound wave detection, and Bluetooth iBeacon.
  • On-going order status and finished order history.
  • Gift coupon redeeming
  • User information management.
  • Combined friends list, gifting, invitation, and search ability.
  • Store branch information.

And for the assumptions used when building the app:

  • Work with a RESTful backend web service based on the API and model referenced in footnote #2 and #3.
  • Most content of the app including the product menu/images, the stores, the award game packages, and the user’s information and friends are dynamically fetched and remotely configurable. Client side is to perform the necessary caching of these data to improve overall user experience.
  • Client is responsible for order configuration and submitting to server. The server is then responsible for order status updates through Apple Push Notification Service (APNS).
  • Giftable friends discovery is to be done on the client side.
  • Promotional award game play status and package are fetched every time the game is entered. Server is responsible for judging the game and maintaining the game states.
  • Once an order is in progress, client is responsible for reporting a stream of location data back to server using the combined means. Server is responsible for the analysis of the location data and calculating the improved estimated time of arrival and optimal queuing of the order.

The design constraints:

  • App is designed for 3.5” or 4” devices running iOS 7 and equipped with location capability.
  • For a typical use case, the app may be put into background after a user has finished submitting an order. So the location tracking service is largely depended upon the ability for the app to continue running in the background while collecting location data. The various iOS background modes and their limits are researched extensively on a section below.

6. Architecture

  • As in most mobile apps, the architecture is driven largely by the front end presentation layer. Since Garffee is a typical UIKit iOS client for interacting with a backend API, it follows the normal MVC[17] approach as suggested by Apple.
  • Automatic reference counting (ARC) is used exclusively in the project except for low level functionalities provided by the CoreFoundation (CF) layer, for which proper manual memory management or Toll-free bridging[18] is required to transfer ownership of objects in the CF space to the Objective-C space. ARC is widely regarded as the preferred way to manage memory in new Cocoa apps.
  • For relational data persistence Apple’s CoreData framework is used. Binary type data and caches are saved directly to the App’s sandbox storage, and a local URL is generated and linked to the associated model.
  • For the networking layer the library AFNetworking and object mapper library RestKit is used, both are built on top of Apple’s native NSURL constructs for HTTP communication.
  • Once data is fetched from the remote API and persisted in the model layer they can be manipulated by the controller classes, the changes will then be reflected in the views.
  • The model layer always keeps the freshest and most reliable master copy of the data, avoid random caching of the entity objects in memory whenever possible to avoid inconsistency between the view and model layers.
  • Use NSFetchedResultsController[19] or Key-Value Observing[20] to properly monitoring changes in the model layer and reflect on the UI.
  • Whenever a time consuming task needs to be performed outside of the main UI thread, use Apple’s Grand Central Dispatch[21] (GCD) for dispatching a job to the background dispatch queue. It uses less resource than a NSThread and is the recommended way to do threading.

Model Layer:

Since we are using Apple’s CoreData as the persistence framework, the model layer is described in the .xcdatamodeld file. Most of the entities are replication of the remote backend counterpart documented in footnote #3. All of the entity NSManagedObject subclasses are located under /BackEnd/Model in the project directory structure. All of the model class have a single character “M” prefix. The following is a diagram documenting the current model - the entities, their properties, and their relationships.

The MUserInfo is the entity responsible for storing the app user’s information. It is the parent entity of both MUserFacebookInfo and MUserAddressBookInfo, which means that the latter two inherits all the properties from MUserInfo. And when fetching all MUserInfo the records for the latter two entities will be returned as well. For pure Facebook or address book contacts their “appID” property should be nil. Once a contact is discovered by the API to be also an app user their “appID” will be filled. So we can determine they are the “giftable” friends. Other types of arrows in the diagram indicates the one-to-one, one-to-many, and inverse relationships between entities.

Besides the properties persisted in the model, these entity classes can be overloaded to provide derived properties. For example in the product entity MProductInfo, a derived property “resolvedImageURL” is implemented as a getter to return the fully resolved image URL relative to the remotely configurable BlobHostName:

@property (nonatomic, readonly) NSString *resolvedImageURL;

- (NSString *)resolvedImageURL {

    return [[MGlobalConfiguration cachedBlobHostName] stringByAppendingPathComponent:self.imageURL];

}

Some of the entity class setters are overridden to ensure data/relationship consistency. For example, in entities that have both an id referencing another entity and a relationship to an object of that entity, like MOrderInfo, both the “userID” property and the “recipient” relationship exists. Ideally both of them should refer to the same logical recipient, and the “userID” should be exactly the recipient’s id; but in real life one or the other could be missing, or the two of them simply are referring to different entities due to bugs. As far as I know, Apple’s CoreData does not provide a way to automatically enforce this type of consistency, so a convention is created to overcome this. In each of the entity class that requires this type of consistency enforcement, a method -(void)changeValue:forKey is added. When the “userID” property is updated, the “recipient” relationship should be updated and vice versa. Then the “userID” and “relationship” setters are overridden to call this method:

- (void)setUserID:(NSString *)userID {

    [self changeValue:userID forKey:@"userID"];

}

- (void)setRecipient:(MUserInfo *)recipient {

    [self changeValue:recipient forKey:@"recipient"];

}

- (void)changeValue:(id)value forKey:(NSString *)key {

    [self changePrimitiveValue:value forKey:key];

    if ([key isEqualToString:@"userID"]) {

        NSFetchRequest *fetchRequest = [MUserInfo fetchRequest];

        fetchRequest.predicate = [NSPredicate predicateWithFormat:@"appID = %@", self.userID];

        NSManagedObject *userObject = [self.managedObjectContext fetchUniqueObject:fetchRequest];

        if (userObject && [userObject isKindOfClass:MUserInfo.class]) {

            [self changePrimitiveValue:userObject forKey:@"recipient"];

        }

    } else if ([key isEqualToString:@"recipient"]) {

        if (self.recipient.appID.length > 0) {

            [self changePrimitiveValue:self.recipient.appID forKey:@"userID"];

        }

    }

}

This works by utilizing Apple’s Key-Value-Coding[22] concept. This method can be seen used in various entity classes to enforce consistency.

To work with CoreData, at least one or more NSManagedObjectContext[23] is needed. In the application native singleton AppDelegate two contexts are initialized - managedObjectContext, which is the main context associated with the application’s main thread and main queue; and persistentStoreManagedObjectContext, which associates with the background threads and queue and saving directly to persistent storage. The two contexts are set up by RestKit. The main context is a child context to the persistent store context, any changes made to the main context by the main thread is not automatically persisted until [context saveToPersistentStore] is called. Most changes triggered by the main thread should write to the main context. For tasks performed in the background the persistent store context can be saved directly. But the persistent store context should be saved sparingly to avoid locking up the UI.

Web service object mapping:

The web service to local object serialization/de-serialization and persistence functionality is provided by the RestKit framework. In order for RestKit to perform the mapping from web service-provided JSON data to local objects or managed objects, a mapping description is needed for each entity. That is done by implementing the protocol RKMappableEntity (for persisted managed objects) or RKMappableObject (for in-memory transient objects):

@protocol RKMappableEntity <NSObject>

@required

+ (RKEntityMapping *)defaultEntityMapping;

+ (RKResponseDescriptor *)defaultResponseDescriptor;

@end

@protocol RKMappableObject <NSObject>

@required

+ (RKObjectMapping *)defaultObjectMapping;

+ (RKResponseDescriptor *)defaultResponseDescriptor;

@end

The protocols are very simple; each conforming entity or object class should implement two static getters - one for returning the defaultEntityMapping and the other for defaultResponseDescriptor. The RKEntityMapping is just a specification of how each key in the response JSON dictionary should be mapped against the object’s property key. This again rely on Apple’s Key-Value-Coding concept. The following is an example from the entity representing user information - MUserInfo:

+ (RKEntityMapping *)defaultEntityMapping {

    RKEntityMapping *mapping =

    [RKEntityMapping mappingForEntityForName:NSStringFromClass(self.class)

                        inManagedObjectStore:[RKManagedObjectStore defaultStore]];

    [mapping addAttributeMappingsFromDictionary:

     @{@"Id":                @"appID",

       @"FacebookId":        @"facebookID",

       @"FirstName":         @"firstName",

       @"LastName":          @"lastName",

       @"CreditBalance":     @"creditBalance",

       @"Email":             @"email",

       @"Sex":               @"gender",

       @"Birthday":          @"birthday",

       @"Phone":             @"phoneNumber",

       @"ProfileImageUrl":   @"profileImageURL",

       @"CreatedDateTime":   @"userCreationDate",

       @"LastUpdatedDateTime": @"userLastUpdatedDate",

       @"CoffeeIconId":      @"coffeeIconID"}];

    mapping.identificationAttributes = @[@"appID", @"facebookID"];

    mapping.valueTransformer = [[RestManager sharedInstance] defaultDotNetValueTransformer];

    return mapping;

}

The code should be quite self-explanatory and the same pattern is shared among different entity classes. The left-hand-side keys are the ones from the web service response JSON dictionary; and the right-hand-side keys are the property names for the entity or object class. identificationAttributes are keys that corresponds to the “primary key” of the mapped managed object. If a managed object having that same key already exists, when de-serializing new data the object is updated instead of being duplicated. This is crucial to not create duplicates in the CoreData persistent store since a web service call is often called multiple times throughout the lifetime of the application. Finally valueTransformer is a set of RestKit provided data type transformer objects for converting serialized data types and native Objective-C data types. For example the DotNetDateFormatter included in the set defaultDotNetValueTransformer can be used to convert our .Net backend serialized DateTime object to native Objective-C NSDate objects. This valueTransformer should be used in all entity classes that requires date data type serialization/de-serialization.

+ (RKResponseDescriptor *)defaultResponseDescriptor {

    return [RKResponseDescriptor

      responseDescriptorWithMapping:[self.class defaultEntityMapping]

                             method:RKRequestMethodAny

                        pathPattern:nil

                            keyPath:nil

statusCodes:RKStatusCodeIndexSetForClass(RKStatusCodeClassSuccessful)];

}

The defaultResponseDescriptor is similarly straight forward. It is basically what gets feed into the final object mapping network operation. It specifies which object/entity mapping to use, which HTTP method (GET, POST etc) and what HTTP status codes to respond to. “keyPath” is the key referencing the targeted JSON dictionary under the root JSON dictionary in case there is another level of indirection in the response. For example, [{“data”: {“key”: “value”, …}}], see [MUserFacebookInfo fetchFriendsResponseDescriptor] for more details.

RestManager:

RestManager is the centralized hub of all the RESTful web service calls. It is implemented as a singleton class. It is responsible for remote fetches and data posting against two APIs - the Facebook OpenGarph API, and our backend API. It utilizes RestKit and its object mapping capability for easier networking, serialization, parsing, and persisting of web service data. All the calls can be categorized into a few groups: fetching, posting, putting (updating), and querying. Fetch usually means a HTTP GET fetching directly from a remote endpoint for a given entity. Post is performing HTTP POST for an entity against the remote endpoint. Put is updating the entity with HTTP PUT. And query usually means other forms of service call that does not relate to directly manipulating an entity, such as querying a set of phone numbers to filter out the set of friends who happens to be using the app.

The RestManager class is documented in RestManager.h and most calls should be quite self-explanatory. Most fetching calls are analogous to mapping a remote endpoint with a RKResponseDescriptor described earlier. All method take in a callback handler object which should implement the protocol RestManagerResponseHandler. On success of the operation, the callback method restManagerService:succeededWithOperation:userInfo: is called. On failure restManagerService:failedWithOperation:error:userInfo: is called instead. The callback will pass back a SEL selector object to indicate the source RestManager call, so the handler can handle all the response cases in one place clearly without needing to implement a bunch of similar delegate methods. “userInfo” is a generic NSDictionary type for passing back overflowing information (such as mapping results) for custom handlings. This generic mechanism greatly simplifies the success/error handling logic from traditional NSURL pattern and make chaining of service calls in functions such as the login flow easier to follow.

The following are some of the example declarations in the class with the standardized method signature:

- (void)fetchAppUserInfo:(__weak id<RestManagerResponseHandler>)handler;

- (void)fetchFacebookFriendsInfo:(__weak id<RestManagerResponseHandler>)handler;

DataFetchManager:

DataFetchManager is similar to RestManager but is responsible for handling local but presumably time consuming data fetching/storing on the background thread, such as querying for local address book contacts and caching product images. It also acts as a wrapper around RestManager for performing its calls while supporting auto-retry for a number of times. Similar to RestManager, on success/failure callbacks will be returned to handler which implements the protocol DataFetchManagerHandler. Methods in DataFetchManager usually takes in a managed object context for the associated fetching/saving. The following is an example call of a RestManager function with the automatic retry count specified:

        [[DataFetchManager sharedInstance] performRestManagerFetch:@selector(fetchAppUserInfo:) retries:3];

One thing to note is that for the handling of queryAddressBookContactsInContext, server side will return a list of filtered phone numbers associating the user’s contacts who happens to also be app user (the “giftable” user”. These phone numbers are then canonicalized and queried against the MUserAddressBookInfo entity. On a match the address book contact will become an app user by settings its “appID”. The process is similar for the Facebook friends user discovery.

SessionManager:

SessionManager is a singleton wrapper around RestManager and DataFetchManager for handling everything related to the login/logout flow and session. There are two straightforward methods which functionality should be self-explanatory:

- (void)loginWithFacebookCredentials;

- (void)logoutAndClearDatabase;

In order to intercept the status of the login/logout process, the delegate class should implement the protocol SessionManagerDelegate. The protocol and the rest of the class is documented in SessionManager.h. SessionManager is itself a handler of RestManager, and the whole login flow and handling is implemented quite clearly in the class’s RestManager success handler restManagerService:succeededWithOperation:userInfo:. In summary, the login stages are as listed in the following enum. When a stage is completed, a normalized progress is reported back to the SessionManagerDelegate implementer:

typedef enum {

    SessionManagerLoginStageFacebookAppUser = 1,

    SessionManagerLoginStageAppUser = 2,

    SessionManagerLoginStageFacebookFriends = 3,

    SessionManagerLoginStageAppConfigurations = 4,

    SessionManagerLoginStageAppProducts = 5,

    SessionManagerLoginStageAppStoreBranches = 6,

    SessionManagerLoginStageAppOrderHistories = 7,

    SessionManagerLoginStageAppGiftCoupons = 8,

    SessionManagerLoginStageAppProductImages = 9,

    // give more weight to the product image fetching stage

    SessionManagerLoginStageTotal = 12

} SessionManagerLoginStage;

Location Tracking:

CoreLocation:

The TimeTracker class contains code to demonstrate the use of the CoreLocation framework. CLLocationManager is the main object for interacting with the API. There are parameters that can be configured for the locationManager object at initialization time:

_locationManager.desiredAccuracy = kCLLocationAccuracyHundredMeters;

_locationManager.distanceFilter = 100;  // meters

_locationManager.pausesLocationUpdatesAutomatically = YES;

_locationManager.activityType = CLActivityTypeAutomotiveNavigation;

When location is updated, the CLLocationManagerDelegate callback method locationManager:didUpdateLocations:

will be called. Currently the class is implemented such that the updated location data is persisted in the MapTrackingAnnotation entity and reflected as dropped pins on a MapView which is accessible through Settings -> Debug. Once the location data reporting remote API is ready this class can be augmented to report the set of MapTrackingAnnotation’s.

Some other methods in this class worth commenting are:

- (void)scheduleInBackground

For firing a timer right when the app is about to enter background (when user presses the home button or the device went to sleep). The timer is scheduled currently to be fired periodically every 60 seconds to update the location.

- (void)updateInBackground

This method is called by the background firing timer to update and log and the current location.

- (void)handleBackgroundFetchWithCompletionHandler:code:

This method is called in turn by various background event callback handlers such as: didReceiveRemoteNotification: for handling push notification events, and performFetchWithCompletionHandler: for handling a system generated background fetch wake events. The only thing that this method does is firing a background task for extending the background running time (for limited time background modes) and update and log the current location.

All of these measures are done to maximize the chance and time to run code in the background for the purpose of updating the user location. Server can choose to send a “silent push notification” (like a keep-alive pulse) to periodically wake up the client in the duration of an on-going order to keep the location data fresh. For more information, see the Background Modes section below.

Soundwave Detection:

Soundwave detection is a novel idea for performing short-distance, preferably indoor locationing. It works by using tone-generating sources (or simply speakers) to emit a unique human-inaudible high frequency sound wave and detecting it in app through the device’s microphone.

The feature is encapsulated in three classes: SoundwaveRecorder, SoundwaveAnalyzer, and SoundwaveTestViewController. SoundwaveRecorder is responsible for recording buffered raw sound data using Apple’s CoreFoundation API Audio Queue Services[24]. There are two straightforward methods: startRecording and stopRecording. When the buffered audio data is ready the method soundwaveRecorder:didReceiveAudioData: is called on the handler implementing SoundwaveRecorderDelegate to hand over the audio data. SoundwaveRecorder is set up with a 44100 Hz sample rate, 32-bit bit rate, and buffer size to make working with the SoundwaveAnalyzer easier.

SoundwaveAnalyzer is responsible for the digital signal processing work. It takes in data from SoundwaveRecorder and perform a Fast Fourier Transform (FFT)[25] to convert the audio data from time domain to frequency domain to enable frequency analysis. This class uses the low-level Accelerate library for hardware accelerated vector computations. Below are the descriptions for the methods in this class:

- (void)initializeFFT:(UInt32)numOfFrames;

Initializes the FFT component with a working size.

- (BOOL)computeFFT:(NSData *)inSamples;

Applies a windowing filter function and computes FFT for the incoming data to prepare working data for the frequency detection methods.

- (FrequencyInfo)findTargetFrequencyLow:(Float32)freqLow high:(Float32)freqHigh;

Find the dominant frequency (one with highest magnitude) within the frequency range specified in Hz. A FrequencyInfo struct is returned specifying the target frequency bin and its actual magnitude.

- (FrequencyInfo)findDominantFrequency;

Find the dominant frequency through the whole analyzed spectrum.

SoundwaveTestViewController is the middle man between the recorder and analyzer. It is responsible for coordinating the recorded data, process and spectrum-analyze it, and visualizing the result on a heat map. The test view is accessible through Settings -> Debug. The delegate method soundwaveRecorder:didReceiveAudioData: contains example of how to compute FFT on the audio data in background and filtering the result by normalizing and applying a threshold value for a good presentation. When a frequency is considered to be a positive signal based on the parameters, the FrequencyInfo is logged and persisted to the MFrequencyInfo entity.

More details about this feature and its test plan is documented in:

https://docs.google.com/a/nbition.com/document/d/1ddl00BIlZGwkUwuOIAP_CC4TViziShmbnnMUnRYFwnM/edit?usp=sharing

Bluetooth iBeacon:

Bluetooth low-energy iBeacon is a new API released by Apple to address the proximity locationing and near field communication (NFC) problems in iOS. Devices capable of Bluetooth low-energy (e.g. smart devices, aftermarket sensors[26], etc) can be configured to broadcast a beacon signal associating a UUID. Other devices can then detect these signals within a proximity and act as short-distance location tracking. The usage of this feature is demonstrated in BluetoothBeaconTestViewController. It is configured with a UUID and able to both broadcast and receive the signal. The class implements the delegate method locationManager:didRangeBeacons:inRegion: and can detect 4 levels of proximity based on the API: ProximityUnknown, ProximityFar, ProximityNear, and ProximityImmediate. It is basically coordinating the roles as CBPeripheralManagerDelegate and CLLocationManagerDelegate and receiving delegate callbacks on locationManager. The results will then be reflected on the UI. The whole class should be rather easy to follow. The test view is accessible through Settings -> Debug.

Background Modes:

In order for the app to continue update location in the background. Apple provides several background modes for special-purpose apps to temporary wake up while the app is put into suspended state (i.e. the UIBackgroundModes):

  • location - App needs to update location in background:
  • When location is available to app, can use combination of NSTimer and turning off/on LocationManager to schedule a periodic execution in background indefinitely (tested with timer firing every 1 minute in background for ~30 minutes).
  • Can be turned off by user.
  • Plus the usual region monitoring and significant location change services for waking up the app (for 10 seconds or so from what I heard).
  • voip - app needs voip functionality and need to keep a connection alive for receiving incoming calls:
  • Need to set request as the VoIP type, also need to maintain a chained long poll connection with server to keep the background execution flowing (tested with a server script holding each connection for 1 minute, resulting in a 1 minute poll interval in app).
  • Without the voip background mode on, wrapping the chained long poll connections with UIBackgroundTask has no effect, app goes to suspended state after first background task is ended.
  • keepAliveHandler (fire every 10 minutes in background) can only be used when this mode is set.
  • Declaring voip in non-VoIP app will almost certainly get rejected by App Store.
  • fetch - App needs to fetch data in background:
  • Can’t control when the fetching handling fire in background.
  • Empirically fires every 15 minutes or so in background given that the phone is full powered, but can’t guarantee, exact schedule depends on usage pattern, battery life, network conditions as Apple claims.
  • Once woken up, have about 30 - 40 seconds of execution time then immediately put back to suspended state.
  • New in iOS7.
  • Can be turned off by user.
  • remote-notification - App needs to fetch data in background after receiving a push notification:
  • Sending silent push notifications: {“aps”:{“content-available”:1,’sound’:0}} with NewsstandContentAvailability mode, thus waking up the handler without user’s notice.
  • Should be ok to use server-induced APNs to keep the app’s heart beat in background (tested with 1 minute push interval for 30 minutes).
  • Silent notifications may have rate limiting imposed by Apple.
  • Once woken up, have about 30 - 40 seconds of up time.
  • New in iOS7.
  • Can be turned off by user.
  • audio - for playing back or recording audio session in the background:
  • Allows the audio recording session to be continued in background.
  • Or plays a silent sound track in background to allow background time.
  • Not recommended, almost certainly will get rejected by App Store.
  • The recording session can be interrupted, or user can override what’s playing in background by switching app.
  • No background modes available at all (e.g. all turned off by users):
  • ~3 minutes of execution time by using UIBackgroundTask immediately after application enters background.
  • Then immediately suspended indefinitely, no 10 minute keepAliveHandler.

The background modes that are being used currently are: location, fetch, remote-notification, and audio.

User Interface Layer:

  • The interface is implemented according to the wireframe documented in footnote #1. The front end constitutes the bulk of the codebase.
  • The UI control flow is mainly achieved by laying out on Storyboard in the Interface Builder. Standalone view controllers and some components such as custom table view cells can be drawn directly onto the storyboard. Because of the non-trivial control flow the Segue functionality of Storyboard is not fully utilized. Most of the view controllers in the Storyboard are still instantiated and presented through code.
  • For the reusable or more complex views and table view cells draw them in traditional XIB files, and load them into the view dynamically through code. The convention is to draw and layout views using the Interface Builder (Storyboard or XIBs) as much as you can, leaving the more dynamic parts to code.
  • Separate different front end concerns to well-modularized classes. For native UI components such as UINavigationController that need customization or theming, extends the native class with the keyword “TS” prepended, e.g. TSNavigationController and do the customization in the class. All theming and UI utility functions should be put under Utilities/TSTheming. All the related classes for views, view controllers, and IB files should be put under FrontEnd in the project structure.
  • For other general UI practice, consult the existing code and follow Apple’s guideline and convention as much as possible.

10. Security

Although security is not the main issue revolving around this mobile client product. Some security measures are necessary to protect the user’s sensitive information and to avoid abusing of the app and the system:

SecurityManager:

SecurityManager is responsible for storing/reading sensitive data securely to Apple’s Keychain service. The data which considered to be sensitive currently includes the “AppToken”, which is the authentication token used for communications with our backend web service. The Facebook API auth token is managed securely by the Facebook SDK itself. Apple’s Keychain[27] service is a rather low-level API built on top of the CoreFoundation[28] (CF) framework. Which has a C-style programming pattern. The whole SecurityManager’s interface is listed as follow:

static NSString *SecurityManagerIdAppToken  = @"AppToken";

+ (NSData *)searchKeychainCopyMatching:(NSString *)identifier;

+ (BOOL)createKeychainValue:(NSString *)value forIdentifier:(NSString *)identifier;

+ (BOOL)updateKeychainValue:(NSString *)value forIdentifier:(NSString *)identifier;

+ (void)deleteKeychainValue:(NSString *)identifier;

It basically abstracts the Keychain service to the Objective-C level and lets user easily search/create/update/delete a keychain item based on an identifier. The identifier is used to initialize a search dictionary for locating the keychain item. Apple Keychain service framework is then responsible for encrypting/decrypting the data securely on a per app (or rather, per provisioning profile used for signing the app) basis.

Other forms of security measures necessary includes:

  • Enable HTTPS communication against the server backend on production to avoid any middle man from eavesdropping the API traffic and discern sensitive information such as the app token. The traffic will then be signed and encrypted with Apple’s default credentials.
  • Protecting user’s credit balance and bullet-proofing all the API interactions with server to maintain an integral credit balance and avoid abusement.
  • Secure storage of app user’s personal information.
